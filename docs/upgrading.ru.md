# Обновление до новой версии

[English](upgrading.md)

Данные (сотрудники, курсы, результаты тестов, секреты интеграций и уведомлений) живут в трех
именованных Docker-томах, объявленных в `docker-compose.yml`: `pg_data` (база Postgres),
`media_data` (загруженные картинки в главах) и `static_data`. Обновление образов их не трогает -
том пересоздается только если его явно удалить. Отсюда главное правило.

## Единственное по-настоящему опасное действие

**Никогда не выполняйте `docker compose down -v` и не удаляйте тома командой `docker volume rm`
на боевом окружении.** Флаг `-v` удаляет все три тома вместе с базой безвозвратно. Обычный
`docker compose down` (без `-v`) безопасен - тома остаются на диске.

## Обычное обновление

1. Посмотрите [CHANGELOG.ru.md](../CHANGELOG.ru.md) - если у версии, на которую обновляетесь, есть
   раздел "Важно при обновлении", сначала выполните описанные там ручные шаги (например, новую
   переменную окружения).
2. Сделайте бэкап базы (см. ниже) - это на всякий случай, а не потому что обновление обычно
   что-то ломает.
3. Заберите новый код и пересоберите:
   ```bash
   git pull
   docker compose up -d --build
   ```
   При старте контейнер `backend` сам применяет накопившиеся миграции базы
   (`python manage.py migrate`) и собирает статику - руками ничего катать не нужно.
4. Проверьте версию в подвале меню консоли или в шапке Django Admin - она должна совпадать с той,
   что вы разворачивали.

Если после пересборки `backend` API отвечает 502 через фронтенд-nginx, а логи backend при этом
чистые - перезапустите фронтенд: `docker compose restart frontend`. Это не потеря данных, просто
nginx закэшировал IP старого контейнера backend при пересборке.

## Бэкап и восстановление базы

Бэкап перед обновлением:

```bash
docker compose exec db pg_dump -U awareness awareness > backup-$(date +%Y-%m-%d).sql
```

Восстановление (если что-то пошло не так - сначала откатите код на предыдущую версию, потом базу):

```bash
git checkout v1.0.0   # предыдущий тег, на который откатываетесь
docker compose up -d --build
cat backup-2026-07-07.sql | docker compose exec -T db psql -U awareness awareness
```

## Версионирование

Проект следует [Semantic Versioning](https://semver.org/lang/ru/) и тегирует релизы в git
(`v1.0.0`, `v1.1.0`, ...). Разработка ведется в `main`, а тег - это конкретная зафиксированная
версия. Для боевого окружения разумно разворачиваться не с "плавающего" `main`, а с конкретного
тега:

```bash
git fetch --tags
git checkout v1.0.0
docker compose up -d --build
```

Список изменений по каждой версии - в [CHANGELOG.ru.md](../CHANGELOG.ru.md). Если версия ломает
обратную совместимость (переименован обязательный параметр окружения, нужна ручная миграция
данных и т.п.), это MAJOR-версия (первая цифра), и в CHANGELOG у нее обязательно будет раздел с
описанием, что сделать руками перед обновлением.
